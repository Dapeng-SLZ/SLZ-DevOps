# 产品化与功能填充路线

## 阶段 1: 从可部署到可运营

当前仓库已经完成这一阶段的大部分基础工作：

- openEuler 上默认部署可用
- 监控、日志、告警、AI 最小分析链路可用
- CMDB、Neo4j、Tempo 已具备扩展底座
- 前端控制台 apps/console 已作为独立界面骨架接入

这一阶段剩余建议：

- 把控制台接入真实 Prometheus 查询结果与告警统计
- 将作业中心从受控模板执行升级为脚本库、批量执行和回滚模板
- 为平台业务库补充迁移脚本、保留策略和定时备份任务
- 为平台业务库补充恢复演练流程和异地备份策略

## 阶段 2: 从基础监控到平台控制面

这一阶段建议补齐真正的平台管理能力：

- 事件中心：告警收敛、确认、抑制、升级、关闭
- 作业中心：脚本库、批量执行、回滚模板、执行审计
- 变更中心：发布记录、配置变更、变更窗口和审批流
- 资产中心：主机、服务、应用、环境和标签体系
- 权限中心：账号、角色、菜单、租户和环境隔离

建议将这一阶段进一步按控制台导航域拆解实施：

- 第一批：平台总览、事件墙、任务中心、CMDB 服务管理
- 第二批：主机资产、定时任务、多云账号与资源同步
- 第三批：工单系统、容器管理、中间件巡检
- 第四批：统一权限、菜单配置、租户与环境隔离

## 阶段 3: 从规则分析到智能运维

这一阶段建议把 AI 能力从示例接口升级为真实平台能力：

- 时序异常检测模型持久化与任务化执行
- 告警与日志的跨源事件关联
- 基于 CMDB 与 Trace 的拓扑传播分析
- 根因候选排序与置信度解释
- 标准自愈动作编排与受控执行

这一阶段的控制台体验建议参考“内嵌智能助手”模式：

- 助手可直接读取平台上下文，而不是做独立聊天页。
- 助手输出优先生成任务、巡检、诊断建议和执行确认单。
- 高风险动作必须进入“执行需确认”流程，避免直接自动执行。
- 聊天记录、执行记录和审计日志统一落到平台业务库。

## 阶段 4: 从单机版到发行平台

这一阶段关注交付形态和企业落地：

- 拆分 api-gateway、job-runner、event-center 等服务
- 增加 PostgreSQL 或 MySQL 作为平台业务库
- 输出 RPM、离线安装包和升级脚本
- 增加多节点部署、备份恢复和高可用方案

## 推荐的前后端拆分结构

如果目标是“自建平台”而不是“脚本集合”，建议按下面的结构推进：

- apps/console: 前端控制台，面向运营与管理界面
- services/api-gateway: 统一 API 出口、鉴权、聚合与 BFF
- services/ai-engine: 智能分析服务
- services/cmdb: 资产与拓扑服务
- services/event-center: 告警与事件生命周期服务
- services/job-runner: 自动化作业与执行服务

如果继续向你提供的参考架构靠拢，建议补出以下服务：

- services/api-gateway: 控制台统一入口、鉴权、聚合和菜单编排
- services/auth-center: 登录、会话、角色、权限、租户
- services/agent-assistant: 智能助手编排、工具调用、执行确认
- services/host-center: 主机资产、主机任务、定时任务
- services/ticket-center: 工单、审批、变更、发布记录

## 结论

前端看板和管理界面完全可以单独开发，并且应该单独开发。

Grafana 更适合做观测看板，apps/console 更适合承载平台管理、流程控制和业务编排。两者并行存在，才是自建运维平台更稳妥的产品形态。
