# 架构说明

## 1. 架构定位

本发行版不是单纯的监控工具集合，而是一个面向 openEuler 24.03 LTS SP3 的 AIOps 平台基础底座。它把观测、告警、分析、自动化四部分先串成一个可运行闭环，再逐步扩展到知识图谱、CMDB、容量预测和自愈执行。

## 2. 组件边界

- Prometheus：负责采集平台指标与规则告警。
- Alertmanager：负责告警路由与聚合。
- Grafana：负责统一展示。
- Loki + Promtail：负责日志聚合。
- Blackbox Exporter：负责基础探测。
- Node Exporter：负责主机资源采集。
- Tempo：负责链路追踪存储与查询。
- CMDB：负责服务元数据、依赖关系和责任边界。
- Neo4j：负责图关系与后续知识图谱扩展。
- AIOps Engine：负责分析接口、Webhook 汇聚、根因分析与后续 AI 扩展。
- Console：负责自建前端控制台、平台管理界面与业务控制面。

## 3. 落地原则

- 优先单机可用，再逐步走向多节点与高可用。
- 优先开源基础设施，降低首期交付复杂度。
- 优先兼容 openEuler 默认系统能力，减少额外依赖。
- 优先通过容器化和脚本化降低实施门槛。
- 优先前后端分层，避免把平台管理能力全部堆叠在 Grafana 中。

## 4. 后续扩展建议

- 继续扩展 deploy 目录下的分析与数据模块，并补齐高可用部署方案。
- 将 AI Engine 拆分为 detection、correlation、remediation 三个服务。
- 把脚本交付升级为 RPM 包或离线安装包，形成正式发行件。
- 以 apps/console 为起点构建独立控制台，并逐步引入统一 API 网关。

## 5. 控制台参考架构

后续控制台整体设计可参考当前提供的目标形态，采用“登录门户 + 左侧导航 + 工作台 + 智能助手”的四层结构。

建议的信息架构如下：

- 登录与门户：统一品牌入口、账号登录、工作台进入、公告与产品能力概览。
- 工作台：承载首页总览、待办、巡检、值班、事件摘要和近期作业记录。
- 左侧业务导航：按领域拆分菜单，而不是按底层组件拆分菜单。
- 智能助手：作为控制台内嵌能力，负责问答、诊断建议、作业生成和确认执行。

建议优先落成的一级导航域如下：

- CMDB：资产、服务、应用、依赖、责任人、标签。
- 主机中心：主机资产、任务中心、定时任务、主机申请。
- 多云管理：云账号、区域、资源同步、配额与成本视图。
- 工单系统：变更、发布、审批、执行记录。
- 容器管理：集群、命名空间、工作负载、镜像、发布。
- 中间件：Redis、MySQL、MQ、Nginx 等组件巡检与变更。
- 可观测性：平台总览、监控看板、日志中心、链路追踪、告警中心。
- 事件墙：事件总览、事件流转、抑制、升级、通知编排。

这意味着当前仓库里的 Grafana、Prometheus、Loki、Tempo 继续作为底层能力存在，而 apps/console 逐步上移为统一工作台和控制面。
